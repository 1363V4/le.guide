i'll argue that using pyglet is **good for you**.
that it is better than using a fully-fledged game engine.

because there's a trick.

you'll get your game out there faster.
but it's gonna be a different game.

you'll make a better game.
but it's gonna be a different game.

you'll enjoy working on your game more.
but it's gonna be a different game.

because you are going to use
**data-oriented design**.

---

there's this joke that every game
is homeomorphic to excel.

it's kinda true.

any game,
if you *dry it down*,
is data manipulation.
this data may trigger mechanisms,
but those mechanisms only occur to you
by changing data you see.

data-oriented design
(data-driven development, whatever)
is just starting from this dry substrate,
which is easier to reason about and optimize,
before *juicing it* back up.

> juice: the process of adding fluff to a game
> (particles, camera effects, feedbacks, sounds...)
> to make it more enjoyable

devs love to start with *some* juice.
and rightly so!
it's useful to stay motivated!
they make a little sprite character,
make him bounce around,
some animations...
and start making the mistake of thinking of it
like an object.

the unfortunate consequence is they then
want to add "mechanics" and spend a lot of time wrangling them,
instead of adding juice.

and in the end, there's not enough juice.
you'll need way more than you think anyway.

data-oriented design proposal
is that you'll have a better dev experience
working on a boring excel shell first,
before moving to the juice.
